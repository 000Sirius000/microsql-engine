# ADR-0001: Складний рушій фільтрації даних через Specification

## Статус

Прийнято.

## Контекст

Поточний Micro-SQL Engine працює з CSV-файлами та підтримує базові SQL-подібні запити з `SELECT`, `FROM`, `WHERE` і `ORDER BY`. Проблема попереднього підходу полягає в тому, що фільтрація рядків у `WHERE` була прив'язана до конкретних класів виразів і простих логічних комбінацій. Через це розширення умов, наприклад:

```sql
WHERE (age > 20 AND role = 'admin') OR (salary > 5000)
```

ускладнює підтримку коду, бо кожна нова операція може вимагати змін у вже наявній логіці виконання запиту.

Потрібно зробити фільтр таким, щоб:

- складні логічні ланцюжки будувались як дерево незалежних об'єктів;
- кожна умова мала єдиний контракт перевірки рядка CSV;
- нові типи умов можна було додавати без зміни основного ітератора в `engine.py`;
- старі запити та старі тести продовжували працювати.

## Рішення

Для нового рушія фільтрації обрано патерн **Specification**.

Створено абстракцію `ISpecification<Row>` з методами:

- `is_satisfied_by(row)` / `IsSatisfiedBy(row)` — перевірка рядка;
- `And(spec)` — логічне `AND`;
- `Or(spec)` — логічне `OR`;
- `Not()` — логічне заперечення;
- `collect_identifiers()` — збір колонок, які треба перевірити перед виконанням.

Парсер `WHERE` тепер будує дерево специфікацій:

```python
AndSpecification(
    GreaterThanSpec("age", 20),
    EqualsSpec("role", "admin"),
)
```

Для SQL-виразу:

```sql
(age > 20 AND role = 'admin') OR (salary > 5000)
```

парсер формує дерево `OrSpecification`, яке містить два вкладені піддерева. `engine.py` не знає деталей конкретних умов. Він лише викликає `query.where.is_satisfied_by(row)` для кожного рядка.

Додатково збережено сумісність зі старою моделлю `Comparison`, `Logical` та методом `evaluate(row)`, щоб regression-тести попередньої лабораторної роботи не потрібно було змінювати.

## Наслідки

### Позитивні наслідки

- Дотримано **OCP (Open/Closed Principle)**: нові умови можна додавати через нові класи специфікацій без переписування `engine.py`.
- Логіка фільтрації відокремлена від завантаження CSV, сортування та формування результату.
- Підтримуються вкладені дужки, `AND`, `OR`, `NOT` і пріоритет операторів.
- Старі тести залишаються валідними, бо `Comparison` і `Logical` працюють як специфікації.
- Перевірка невідомих колонок залишилась централізованою через `collect_identifiers()`.

### Компроміси

- Парсер став трохи складнішим, бо тепер він будує дерево об'єктів Specification, а не просто старе дерево виразів.
- У коді залишено сумісні назви `Expr`, `Comparison`, `Logical`, щоб не ламати попередній API. Це створює невеликий перехідний шар, але він корисний для regression testing.

### Вплив на систему

- `engine.py` отримав мінімальну зміну: замість `evaluate(row)` основний шлях використовує `is_satisfied_by(row)`.
- `parser.py` відповідає за побудову дерева специфікацій.
- `tokenizer.py` додатково розпізнає `NOT`.
- Додано `specifications.py` як публічний модуль для роботи з патерном Specification.
- Додано `config.py` і JSON-конфігурацію для безпечного ввімкнення параметрів фільтрації під час старту програми.

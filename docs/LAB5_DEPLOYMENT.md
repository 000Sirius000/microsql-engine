# Лабораторна робота №5

## Тема

Розгортання. Контейнеризація та CI/CD.

## Мета роботи

Метою роботи є отримання практичних навичок контейнеризації консольного застосунку, налаштування безперервної інтеграції та автоматизації створення релізних артефактів.

У межах роботи було виконано:

- контейнеризацію Python-застосунку за допомогою Docker;
- створення `.dockerignore`;
- налаштування CI/CD pipeline за допомогою GitHub Actions;
- автоматичну перевірку коду лінтером;
- запуск unit-тестів;
- збірку Python wheel;
- збірку Docker-образу;
- автоматичне створення релізу при створенні Git-тегу.

## Опис проєкту

MicroSQL Engine — це консольний Python-застосунок для виконання спрощених SQL-подібних запитів до структурованих текстових даних, зокрема CSV-файлів.

Проєкт використовує такі інструменти:

- Python;
- Poetry;
- Ruff;
- Pytest;
- Docker;
- GitHub Actions.

## Контейнеризація застосунку

Для контейнеризації застосунку було створено файл:

```text
Dockerfile
```

Dockerfile розміщено в корені репозиторію.

У ньому використано багатоетапну збірку, тобто multi-stage build. Це дозволяє відокремити середовище збірки від середовища виконання.

### Структура Dockerfile

Dockerfile складається з двох основних етапів:

1. `builder` — етап збірки;
2. `runtime` — етап виконання.

На етапі `builder`:

- використовується образ `python:3.11-slim`;
- встановлюється Poetry;
- копіюються файли проєкту;
- виконується збірка Python wheel.

На етапі `runtime`:

- використовується легковаговий образ `python:3.11-slim`;
- копіюється готовий wheel з етапу `builder`;
- встановлюється зібраний пакет;
- задається точка входу для запуску утиліти `microsql`.

Такий підхід дозволяє не включати у фінальний Docker-образ зайві інструменти збірки.

## Файл .dockerignore

Також було створено файл:

```text
.dockerignore
```

Він використовується для виключення зайвих файлів із Docker build context.

До `.dockerignore` додано:

- `.git`;
- `.venv`;
- кеші Python;
- кеші Pytest;
- кеші Ruff;
- файли IDE;
- локальні логи;
- результати попередніх збірок;
- тимчасові файли.

Це зменшує розмір контексту збірки та робить процес створення Docker-образу чистішим.

## Збірка Docker-образу

Для збірки Docker-образу потрібно перейти в кореневу папку проєкту та виконати команду:

```bash
docker build -t microsql-engine:local .
```

Пояснення команди:

- `docker build` — команда для створення Docker-образу;
- `-t microsql-engine:local` — назва образу та його тег;
- `.` — поточна папка використовується як контекст збірки.

Після успішної збірки створюється локальний Docker-образ:

```text
microsql-engine:local
```

## Запуск контейнера

Для запуску програми всередині контейнера використовується команда:

```powershell
docker run --rm -v "${PWD}:/data" microsql-engine:local /data/query.sql --data-dir /data --config /data/microsql.config.json
```

У цій команді поточна папка проєкту монтується всередину контейнера як директорія `/data`.

Це потрібно для того, щоб контейнер міг отримати доступ до вхідних файлів:

- `query.sql`;
- CSV-файлів;
- `microsql.config.json`.

## Налаштування CI/CD

Для автоматизації перевірки коду та збірки проєкту було налаштовано GitHub Actions.

Файл workflow розміщено за шляхом:

```text
.github/workflows/ci-cd.yml
```

Pipeline запускається автоматично при:

- push у гілку `master`;
- створенні Pull Request у гілку `master`;
- створенні Git-тегу версії у форматі `v*.*.*`.

## Етапи Continuous Integration

CI pipeline виконує такі кроки:

1. Checkout репозиторію.
2. Встановлення Python.
3. Встановлення Poetry.
4. Встановлення залежностей проєкту.
5. Запуск Ruff для перевірки коду.
6. Перевірка форматування.
7. Запуск unit-тестів.
8. Збірка Python wheel.
9. Збірка Docker-образу.
10. Тестовий запуск Docker-контейнера.

## Перевірка якості коду

Для перевірки якості коду використовується Ruff.

Команда запуску лінтера:

```bash
poetry run ruff check .
```

Команда перевірки форматування:

```bash
poetry run ruff format --check .
```

Якщо Ruff знаходить стилістичні помилки або неправильне форматування, pipeline завершується зі статусом `failed`.

## Запуск unit-тестів

Для запуску unit-тестів використовується Pytest.

Команда запуску тестів:

```bash
poetry run pytest
```

Якщо хоча б один тест завершується з помилкою, pipeline також отримує статус `failed`.

Це дозволяє автоматично блокувати збірку при наявності помилок у коді.

## Збірка артефакту

Під час виконання pipeline створюється Python wheel.

Команда збірки:

```bash
poetry build
```

Результат збірки зберігається в директорії:

```text
dist/
```

Wheel-файл використовується як релізний артефакт.

## Збірка Docker-образу в pipeline

Окрім Python wheel, pipeline також збирає Docker-образ:

```bash
docker build -t microsql-engine:${{ github.sha }} .
```

Після цього виконується тестовий запуск контейнера, щоб перевірити, що контейнеризована версія застосунку працює коректно.

## Continuous Deployment та реліз

CD-частина pipeline запускається при створенні Git-тегу версії.

Формат тегу:

```text
v*.*.*
```

Приклад:

```bash
git tag -a v1.0.0 -m "Release v1.0.0"
git push origin v1.0.0
```

Після push тегу GitHub Actions автоматично створює GitHub Release.

До релізу прикріплюються такі артефакти:

- Python wheel;
- Docker image у форматі `.tar`.

## Висновки

У результаті виконання лабораторної роботи було налаштовано процес розгортання консольного застосунку MicroSQL Engine.

Було створено Dockerfile з багатоетапною збіркою, що дозволяє запускати застосунок у незалежному контейнеризованому середовищі. Для зменшення контексту збірки було додано файл `.dockerignore`.

Таким чином, проєкт було підготовлено до стабільного розгортання, автоматизованої перевірки та публікації релізів.
